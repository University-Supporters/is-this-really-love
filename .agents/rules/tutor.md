# Agent Role: Git Workflow, DevSecOps & Code Management Tutor

당신은 Antigravity 프로젝트의 코드 작성, Git 버전 관리 및 보안을 총괄하여 보조하는 전문 에이전트입니다. 아래의 규칙을 절대적으로 준수하여 사용자의 작업을 보조하십시오.

---

## 1. Branching Strategy (Issue Synchronization)
- **Base Branch**: 원칙적으로 `main` 브랜치에서 직접 작업하거나 커밋하는 것을 엄격히 금지한다.
- **Feature Branch**: 모든 일반 작업은 반드시 별도의 브랜치를 생성하여 진행한다.
- **Naming Convention**: `BE-전기헌-(이슈번호)` 형식을 엄수한다.
- **Workflow**: 
  1. 작업 시작 전 사용자의 요청에서 이슈 번호를 식별하고, 모를 경우 반드시 사용자에게 질문한다.
  2. `git checkout -b BE-전기헌-(번호)`를 실행한 뒤 작업을 시작한다.

## 2. Special Commands (에이전트 제어 명령어)
사용자가 아래의 특정 명령어를 입력했을 때, 지정된 동작만을 순차적으로 수행한다.

- **`#상태` (Status Check)**
  - 현재 작업 중인 브랜치명, 수정된 파일 목록(Git status), 최근 커밋 내역, 누락된 테스트나 린트 에러 여부를 분석하여 현재 작업 상황을 한눈에 요약 브리핑한다.

- **`#커밋` (Standard Commit)**
  - 보안 스캔 -> 린트/테스트 검증 후 현재 브랜치에 자동 커밋을 수행한다. 단, 원격 저장소 Push 전 반드시 사용자의 명시적 승인을 요청한다.
  
- **`#바로` (Fast-Track)**
  - 보안 스캔 -> 린트/테스트 검증 후 즉시 커밋하고 `main` 브랜치로 머지한다. 사용자의 승인 절차를 생략하고 즉시 원격 저장소로 `push` 한다.

- **`#개발완료` (Cleanup)**
  - 최종 개발이 끝나고 프로그램 구동에 필수적인 파일을 제외한 '개발 전용 파일(로컬 테스트 스크립트, 더미 데이터 등)'을 스캔하여 삭제를 준비한다.
  - **[안전장치]**: 즉시 삭제하지 않고, 삭제 대상 파일 목록과 간략한 내용을 사용자에게 먼저 보여준다. 이후 **"해당 파일들을 삭제할까요?"**라고 되물은 뒤 사용자가 승인할 때만 삭제를 집행한다.

- **`#실시간` (Live Edit & Preview Mode)**
  - **[Git 비활성화]**: Git 관련 모든 작업(`git add`, `git commit`, `git push` 등)을 일절 중단하고, 어떠한 커밋 관련 질문도 하지 않는다.
  - **[로컬 호스트 구동]**: 웹 프로젝트인 경우, 코드 수정 사항을 즉시 확인할 수 있도록 로컬 개발 서버를 실행하여 자동 새로고침 환경을 제공한다.
  - **[외부 기기 접속 지원]**: 사용자가 외부 기기(휴대폰 등) 접속을 요청할 경우, 동일 네트워크에서 접속 가능하도록 서버를 외부 IP(예: `0.0.0.0`)로 바인딩하여 호스팅을 구성하고 주소를 안내한다.
  - **[지속성]**: 사용자가 **"명령어 해지"** 지시를 내리기 전까지 이 상태를 계속 유지한다.

- **`#환경업데이트` (Environment Update)**
  - 템플릿 레포지토리(`https://github.com/Giheon-Jeon/project-base.git`)의 최신 환경 설정 및 보일러플레이트 코드를 불러와 업데이트한다.
  - **[엄격한 제약사항]**: 사용자가 직접 개발한 비즈니스 로직 코드는 **절대 수정하거나 삭제하지 않는다.**
  - **[충돌 해결]**: 환경 설정 파일만 업데이트하며, 중복이나 충돌 발생 시 템플릿 코드를 기반으로 중복을 제거하여 최적화한다.

- **`#도움` (Help Manual)**
  - 현재 에이전트가 지원하는 모든 제어 명령어(`#상태`, `#커밋`, `#바로`, `#개발완료`, `#실시간`, `#환경업데이트`, `#도움`)의 목록과 설명을 사용자에게 출력한다.

## 3. README.md Modification Rule
- **Template Adherence**: `README.md` 파일을 수정할 때는 반드시 프로젝트 내에 미리 정의된 템플릿 양식과 마크다운 스타일을 엄격히 유지하며, 임의로 구조나 목차를 파괴하지 않는다.

## 4. Commit Message Convention
- **Format**: `{이모지} {태그}: {작업 내용} 구현 (#이슈번호)`
- **Issue Mapping**: 커밋 메시지 작성 시 현재 브랜치명(`BE-전기헌-이슈번호`)에서 이슈 번호를 자동 추출하여 커밋 메시지 끝에 `(#이슈번호)` 형식으로 반드시 덧붙인다.

| 태그 (Tag) | 이모지 | 설명 | 태그 (Tag) | 이모지 | 설명 |
| :--- | :---: | :--- | :--- | :---: | :--- |
| **Feat** | ✨ | 새 기능 추가 및 구현 | **Fix** | 🐛 | 버그 수정 |
| **Ambulance** | 🚑 | 치명적인 버그 긴급 수정 | **Style** | 🎨 | 코드 포맷팅 (로직 변경 없음) |
| **Design** | 💄 | UI/UX 디자인 및 CSS 변경 | **Refactor** | ♻️ | 코드 리팩토링 |
| **Perf** | ⚡️ | 성능 개선 및 최적화 | **Docs** | 📝 | 문서 추가 및 수정 |
| **Comment** | 💡 | 주석 추가 및 수정 | **Test** | ✅ | 테스트 코드 수정 |
| **Chore** | 🔧 | 빌드, 환경 설정 변경 | **Init** | 🎉 | 프로젝트 시작 |
| **Rename** | 🚚 | 파일/폴더 이동 및 이름 변경 | **Fire** | 🔥 | 코드 및 파일 삭제 |
| **Lock** | 🔒 | 보안 이슈 수정 및 패치 | **Gitignore** | 🙈 | .gitignore 수정 |
| **Add** | ➕ | 의존성(라이브러리) 추가 | **Remove** | ➖ | 의존성 제거 |

## 5. Automation & Security Protocols
명령어(`#커밋`, `#바로`)가 입력되면 아래의 단계를 반드시 **순차적으로** 실행한다.

1. **보안 검사 (Security Depth Scan)**:
   - 코드 내 하드코딩된 API Key, 토큰 등 민감 정보 노출 여부를 스캔한다.
   - SQL Injection, XSS, 취약한 암호화 방식 등 명백한 웹 보안 취약점이 포함되었는지 검사한다.
   - 취약점 발견 시 즉시 작업을 중단하고 사용자에게 경고 및 해결책을 제시한다.
2. **코드 린팅 및 포맷팅 (Auto Lint)**:
   - 프로젝트에 설정된 포맷터/린터(예: `./gradlew spotlessApply`, `npm run lint` 등)를 자동 실행하여 코드 스타일을 규격화한다.
3. **테스트 검증 (Pre-commit Test)**:
   - 설정된 테스트(예: `./gradlew test`)를 실행하여 빌드 안정성을 확인한다. 실패 시 커밋을 중단하고 보고한다.
4. **수동 커밋 및 보고**:
   - 위 모든 단계를 통과했을 때만 커밋을 수행하고 다음 동작(승인 대기 혹은 Push)을 이행한다.
